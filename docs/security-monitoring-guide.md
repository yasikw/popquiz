# セキュリティ監視運用ガイド

## 概要

このガイドは、日本語学習アプリケーションの包括的セキュリティ監視システムの運用手順を説明します。

## 監視システム構成

### 🔍 リアルタイム監視
- **セキュリティアラート**: 攻撃の即座検知と自動対応
- **CSP違反監視**: Content Security Policy違反の追跡
- **レート制限監視**: 異常なリクエストパターンの検出
- **認証失敗監視**: ブルートフォース攻撃の早期発見

### 📊 定期レポート
- **日次**: セキュリティインシデントサマリー、エラーパターン分析
- **週次**: セキュリティスコア傾向、新脅威パターン検出
- **月次**: 包括的監査、コンプライアンス評価

## アラート設定

### 緊急度レベル

#### 🔴 CRITICAL (緊急)
- **条件**: CSRF攻撃、管理API侵入試行、XSS攻撃成功
- **対応時間**: 即座（5分以内）
- **通知先**: SMS + Email + Slack + 電話
- **自動対応**: 即座IP遮断、緊急ロックダウン

#### 🟠 HIGH (高)
- **条件**: 認証総当たり攻撃、異常アクセスパターン
- **対応時間**: 15分以内
- **通知先**: Email + Slack
- **自動対応**: 段階的レート制限、詳細ログ記録

#### 🟡 MEDIUM (中)
- **条件**: レート制限超過、軽微なCSP違反
- **対応時間**: 1時間以内
- **通知先**: Slack
- **自動対応**: ログ記録、統計更新

#### 🟢 LOW (低)
- **条件**: 一般的なエラー、軽微な異常
- **対応時間**: 24時間以内
- **通知先**: ログのみ
- **自動対応**: 統計記録

## 運用手順

### 1. 日常監視業務

#### 毎朝の確認事項 (9:00 AM)
```bash
# セキュリティダッシュボード確認
curl -H "Authorization: Bearer ${ADMIN_TOKEN}" \
  http://localhost:5000/api/admin/security/dashboard

# 夜間インシデント確認
curl -H "Authorization: Bearer ${ADMIN_TOKEN}" \
  http://localhost:5000/api/admin/errors/stats?period=24h

# アラート履歴確認
curl -H "Authorization: Bearer ${ADMIN_TOKEN}" \
  http://localhost:5000/api/admin/security/alerts?since=yesterday
```

#### 日中の監視活動
- **リアルタイムダッシュボード**: 常時表示、異常時即座対応
- **アラート対応**: 受信後5分以内に初動対応開始
- **インシデント記録**: 全対応内容の詳細記録

### 2. インシデント対応フロー

#### Level 1: 自動対応 (0-5分)
1. **検知**: 監視システムが異常を自動検知
2. **分析**: パターンマッチングで脅威レベル判定
3. **対応**: 事前定義ルールに基づく自動対応実行
4. **記録**: 全アクションの詳細ログ記録

#### Level 2: 人的介入 (5-15分)
1. **通知受信**: DevOpsチームがアラート受信
2. **状況確認**: ダッシュボードで詳細状況把握
3. **対策実行**: 必要に応じて手動対策実行
4. **エスカレーション判断**: 上位レベルへの報告要否判断

#### Level 3: 専門家対応 (15-30分)
1. **専門家召集**: セキュリティ専門家の緊急招集
2. **詳細分析**: フォレンジック分析の開始
3. **対策策定**: 包括的対策プランの策定・実行
4. **影響評価**: システム・データへの影響度評価

#### Level 4: 組織的対応 (30分-2時間)
1. **経営陣報告**: 重大インシデントの経営陣への報告
2. **外部協力**: 外部セキュリティ企業との連携
3. **法的対応**: 必要に応じた法的・規制対応
4. **公表準備**: ユーザー・ステークホルダーへの公表準備

## トラブルシューティング

### よくある問題と対処法

#### 1. CSRF保護が過剰に動作する
**症状**: 正常なリクエストがCSRF違反で拒否される
**原因**: トークンの有効期限切れ、クライアント側の実装不備
**対処**:
```bash
# CSRFトークンの状態確認
curl http://localhost:5000/api/csrf-token

# トークンの更新
curl -X POST http://localhost:5000/api/csrf-token/refresh
```

#### 2. レート制限が厳しすぎる
**症状**: 正常ユーザーがレート制限に引っかかる
**原因**: 制限値の設定ミス、異常トラフィックの誤判定
**対処**:
```javascript
// server/middleware/security.ts の設定調整
export const apiRateLimit = rateLimit({
  windowMs: 60 * 60 * 1000, // 1 hour
  max: 200, // 100 → 200 に増加
  // ...
});
```

#### 3. セキュリティログが多すぎる
**症状**: ログファイルの容量が急激に増加
**原因**: デバッグモードの有効、過剰なログ出力
**対処**:
```bash
# ログレベルの調整
export LOG_LEVEL=error  # debug → error

# ログローテーション設定確認
logrotate -d /etc/logrotate.d/security-logs
```

## 緊急時対応

### 重大インシデント発生時

#### 即座実行項目 (5分以内)
1. **サービス状況確認**: アプリケーションの稼働状況確認
2. **攻撃継続性確認**: 現在も攻撃が続いているか確認
3. **自動対策確認**: 自動対策が有効に機能しているか確認
4. **影響範囲特定**: 被害を受けた機能・データの特定

#### 短期対応 (30分以内)
1. **詳細分析開始**: ログ・メトリクスの詳細分析
2. **追加対策実施**: 必要に応じた手動対策の追加
3. **ステークホルダー連絡**: 関係者への状況報告
4. **復旧計画策定**: システム復旧の具体的計画策定

#### 中期対応 (2時間以内)
1. **根本原因特定**: インシデントの根本原因分析
2. **恒久対策策定**: 再発防止のための恒久対策策定
3. **影響評価完了**: システム・データへの最終影響評価
4. **報告書作成**: インシデント対応の詳細報告書作成
